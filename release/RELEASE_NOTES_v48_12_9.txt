BW Monitor v48.12.9-r4 Release Notes
====================================

R2 hotfix
---------

- Fixed a Python SyntaxError in `_v48129_rule_chip`.
- The R1 source used a nested f-string expression containing escaped quotes, which Python rejects with: `f-string expression part cannot include a backslash`.
- The title attribute is now built in a separate variable before rendering the HTML chip.
- No database schema, Agent payload, API contract, retention policy, Abuse state, or configuration behavior changed.
- Full pre-flight and all regression suites pass after the fix.


Release goal
------------
Make Current Abuse as operationally useful as Top VM while making Abuse
history and cleanup behavior explicit, predictable and production-safe.

- NODE / VM

- CPU

- TX PPS

RAM

DISK

- Write IOPS

Resource visualization
----------------------

CPU:
- Shows Full %
- Shows Core %
- Shows assigned vCPU count
- Full % drives a meter normalized against assigned vCPU capacity

RAM:
- Shows Guest %
- Shows Guest Used GiB
- Shows Host RSS
- Shows Assigned RAM
- Guest Used or Host RSS is normalized against Assigned RAM for the meter
- The currently selected RAM sort mode is highlighted

Meter levels:
- Normal: below 70%
- Warm: 70% to below 85%
- Hot: 85% to below 95%
- Critical: 95% and above

Reason / Severity redesign
--------------------------

MAX ratio stays inside the REASON / SEVERITY cell instead of occupying a
separate column.

The cell now shows:

- Ratio
- Primary Abuse type
- Exact metric and threshold that produced the maximum ratio
- Color-coded rule chips
- Exact configured sustained duration
- Active duration chip: Abusing 35m / 2h 40m / 1d 05h

Colors:
- Network: blue
- CPU: red
- RAM: purple
- Disk: orange
- Active/duration: amber

MAX ratio remains transparent:

- CPU = CPU Full / configured CPU threshold
- Network = current directional metric / configured threshold
- Disk = largest enabled Read, Write, Total or IOPS ratio
- RAM High = current % / threshold
- RAM Low Usable = bounded inverse scale from 1.00x at threshold to 2.00x at 0%

It is not a weighted score and values are not added together.

PPS Peak / Window
-----------------

Each direction shows:

- Peak PPS
- Seconds above threshold in the 300-second Agent window
- Required high seconds from policy

Example:

RX PEAK 220k PPS
270/300s high · need 270s

Admin Abuse data controls
-------------------------

Added an explicit data map:

Current Abuse
- vm_abuse_state

Raw History
- vm_abuse_events

Abuse Events by VM
- vm_abuse_incidents

Clear All Abuse Logs now guarantees:

DELETE FROM vm_abuse_events
DELETE FROM vm_abuse_incidents

Current Abuse remains unchanged by history cleanup.

Clear matching and Clear selected:

- Delete the requested raw event rows
- Rebuild grouped incidents from the remaining raw history

New Reset All Abuse Data action:

- Requires exact text RESET ALL ABUSE DATA
- Deletes current state, raw events and grouped incidents
- Does not delete Node/VM inventory, metrics, Agent state, users or API keys

New per-VM data controls:

- Delete Raw History
- Delete Abuse Events by VM
- Reset Current Abuse + streak
- Requires exact text CLEAR VM ABUSE DATA

-------------------

- Operations table markers exist
- PPS Peak / Window exists
- Color-coded rule chips exist
- Reset-all and per-VM Abuse management endpoints are registered
